This appendix maps the OpenShell on OpenShift PRD to the current upstream state reviewed on May 19, 2026 and re-validated end-to-end on a real OpenShift 4.22 cluster (3-node compact, RHCOS kernel 5.14, CRI-O 1.35) on May 20, 2026. It covers repository docs, public NVIDIA docs, public GitHub issues, public PRs, roadmap-labeled issues, the visible GitHub Projects page, and the install-time gaps captured during the live walkthrough.

The bulletproof install guide that distills the live walkthrough is at [Deploy OpenShell on OpenShift](./openshift-deployment-guide). This appendix preserves the requirement-by-requirement gap accounting.

## Executive Summary

OpenShell has crossed the first Kubernetes enablement threshold: it has a published Helm chart, an in-cluster Kubernetes driver, Agent Sandbox integration, supervisor OCI side-loading, basic network policy for sandbox SSH ingress, Kubernetes resource sizing, gateway mTLS support, OIDC role/scope checks, and OCSF sandbox logs.

OpenShift readiness is not complete. The documented OpenShift path is explicitly evaluation-only: install with TLS disabled, PKI job disabled, fixed UID/fsGroup cleared, and the namespace default service account bound to `privileged`. Kata can be tested through the public API, but the CLI lacks a runtime-class flag. Air-gapped deployment is possible only with careful manual image/chart mirroring and pull-secret workarounds. Multi-team operations should use one gateway per team namespace. Shared-gateway multi-tenancy is not ready because users, sandboxes, and providers are not scoped by owner.

The May 20, 2026 live walkthrough on OpenShift 4.22.0-ec.5 surfaced four additional gaps that the prior doc revision did not account for. The bulletproof install guide is updated; the new gaps are tracked individually below so they show up as work items rather than being silently fixed in the runbook.

| New gap | Surface | Verified workaround |
|---|---|---|
| Sandbox supervisor DNS resolver crashes on auto-derived gateway FQDN under OpenShift's `ndots:5` + four search-domain resolv.conf shape. | Supervisor binary on OpenShift. | Set `server.grpcEndpoint` to the trailing-dot FQDN, e.g. `http://openshell.openshell.svc.cluster.local.:8080`. |
| Documented `securityContext.runAsNonRoot=true` + `runAsUser=null` is incompatible with the `USER nvs` (non-numeric) gateway image and kubelet refuses the pod. | Helm chart values + docs. | Unset `runAsNonRoot` in OpenShift values (`runAsNonRoot: null`) and let the SCC assign UID. |
| Pod Security Admission `enforce: privileged` is required at the namespace label level; the SCC binding alone is not enough on OpenShift 4.22 because the cluster default PSA is `restricted`. | Namespace bootstrap. | Label namespace `pod-security.kubernetes.io/enforce=privileged`. |
| The OpenShift path silently depends on a healthy cluster default StorageClass that supports RWO 1Gi/2Gi PVCs without `storageClassName`. ODF/Ceph backends in HEALTH_WARN block the entire install. | Gateway + per-sandbox PVCs. | Deploy a dynamic local provisioner (Rancher local-path-provisioner) with the SCC + SELinux fixes, then mark it default. |

## M1 OpenShift Readiness

| Requirement | Current status | Evidence | Workaround today | Gap |
|---|---|---|---|---|
| Helm chart ships SCCs | Not implemented | No SCC templates or `openshift.enabled` value exist in `deploy/helm/openshell`. Current OpenShift docs use `oc adm policy add-scc-to-user privileged`. | Pre-create namespace and bind `privileged` SCC to `default` service account. | Add chart-rendered SCCs or documented cluster-admin install bundle. |
| Pod Security Admission ready by default | Not implemented | `oc new-project` leaves namespace at the cluster-default PSA (`restricted` on OpenShift 4.x). The SCC binding alone does not relax PSA in time, so sandbox pods are rejected at admission with `violates PodSecurity ... capabilities.add ...`. | Explicitly label namespace `pod-security.kubernetes.io/enforce=privileged` before installing. | Chart should fail-fast or doc/skill should require the PSA label. The OpenShift `MinimallySufficientPodSecurityStandard` annotation should be respected by every Helm-rendered SA/role binding. |
| Gateway image declares non-numeric `USER` | Empirical install bug | Chart sets `securityContext.runAsNonRoot: true` and the OpenShift docs additionally set `runAsUser: null`. The gateway image's Dockerfile declares `USER nvs`. Kubelet refuses the pod: `container has runAsNonRoot and image has non-numeric user (nvs), cannot verify user is non-root`. | Set `securityContext.runAsNonRoot: null` in OpenShift values; SCC handles non-root. | Either (a) numericize `USER` in the gateway Dockerfile, or (b) document `runAsNonRoot: null` as the canonical OpenShift override. |
| Sandbox supervisor DNS on OpenShift resolv.conf | Empirical install bug | With OpenShift's default `ndots:5` plus four search domains, the supervisor's hyper-util DNS resolver returns `failed to connect to OpenShell server` in <5 ms when the auto-derived gateway FQDN `openshell.openshell.svc.cluster.local` is used. `curl --http2-prior-knowledge` from the same pod succeeds. Trailing-dot FQDN or cluster IP both work. | Pin `server.grpcEndpoint` to the trailing-dot FQDN (`http://openshell.openshell.svc.cluster.local.:8080`). | Either (a) fix the resolver to behave correctly with `ndots:5` + search domains, (b) have the chart helper emit the trailing-dot form by default when the cluster is OpenShift, or (c) document it as a required override. |
| Gateway PVC has configurable storage class/size | Not implemented | `deploy/helm/openshell/templates/statefulset.yaml` hardcodes `volumeClaimTemplates` with `accessModes: ["ReadWriteOnce"]`, `storage: 1Gi`, no `storageClassName`. Falls back to cluster default; unhealthy default storage silently breaks install. | Make a healthy SC the cluster default before install. | Add `gateway.persistence.storageClass`, `gateway.persistence.size`, `gateway.persistence.accessModes`. |
| Sandbox PVCs have configurable storage class/size | Not implemented | `crates/openshell-driver-kubernetes/src/driver.rs` hardcodes `volumeClaimTemplates` with `accessModes: ["ReadWriteOnce"]`, `storage: "2Gi"`, no `storageClassName`. Same fall-back behavior as the gateway PVC. | Make a healthy SC the cluster default before install. | Add Helm or template-level overrides for sandbox PVC class/size. |
| SELinux binary relabeling | PRD item is outdated for current main | Current Kubernetes driver side-loads supervisor through OCI image plus `emptyDir` or ImageVolume. No hostPath binary copy/chcon path is present. | Force `supervisor.sideloadMethod=init-container` on OpenShift. | Still need SELinux/SCC documentation for root/capabilities and Kata scoping. |
| cert-manager integration | Partially implemented | `certManager.enabled`, Issuer, Certificate templates, and values exist. | For official OpenShift evaluation, disable TLS. For experiments, install cert-manager and manually extract `openshell-client-tls`. | OpenShift docs do not yet validate cert-manager mTLS as a supported path. |
| `--runtime-class` CLI flag | Not implemented | Public proto has `SandboxTemplate.runtime_class_name`; CLI create args do not include runtime class. | Use Python SDK or lower-level gRPC request with `runtime_class_name="kata"`. | Add CLI flag and docs. |
| Helm image overrides | Mostly implemented | `image.repository/tag`, `supervisor.image.repository/tag`, and `server.sandboxImage` exist. | Use values file to point at internal registry. | Sandbox image pull secrets are missing as first-class config. |
| Air-gapped mirroring docs | Not implemented for OpenShift | No OpenShift-specific air-gap doc. RPM docs mention air-gap concepts; chart docs list values. | Manually mirror gateway, supervisor, base sandbox, Agent Sandbox, cert-manager, local-path-provisioner, busybox helper, and operator images with `skopeo copy --all`. | Add official mirror matrix and disconnected OpenShift instructions, including the dynamic local-storage provisioner choice. |
| OpenShift + Kata tutorial | Not implemented | Existing OpenShift doc does not mention Kata. | Install sandboxed containers operator, verify `RuntimeClass`, create sandbox through Python SDK. | Add first-class tutorial and CLI support. |
| `--command` flag | Not implemented | CLI supports trailing post-create command, not declarative daemon startup. Driver overwrites `OPENSHELL_SANDBOX_COMMAND` on main. | Post-create `openshell sandbox exec`, or track PR #1326 for template env override. | Add public `SandboxSpec.command` and CLI `--command` if daemon UX is required. |
| Interactive `sandbox create` over port-forward | Behavior gap | Foreground `openshell sandbox create` opens an SSH-over-gRPC TTY through `oc port-forward` and hangs on TTY allocation. Non-TTY `sandbox exec` works. | Use `echo "" \| openshell sandbox exec ...` for verification; expose a Route/Ingress for production interactive use. | Either fix TTY handling over kubectl port-forward, or document Route as a prerequisite for interactive `sandbox create`/`sandbox connect`. |
| GitOps sandbox template | Not supported | Driver watches CRs but external adoption stores `spec: None`; docs do not define a supported manifest. | Use CLI/API-created sandboxes. | Define driver-generated CR schema and gateway adoption semantics. |
| Gateway adopts external CRs | Partial and insufficient | Watch path can create gateway records for existing CRs with a sandbox ID, but does not reconstruct policy/providers/spec. | Avoid external CR creation except for debugging. | Implement full adoption, policy sync, provider injection, and SSH/session binding. |

## Current Workarounds

| Need | Workaround |
|---|---|
| Run on OpenShift today | Use plaintext MVP install with `pkiInitJob.enabled=false`, `server.disableTls=true`, `securityContext.runAsNonRoot=null`, cleared UID/fsGroup, manual `privileged` SCC binding, `pod-security.kubernetes.io/enforce=privileged` namespace label, healthy default StorageClass, and `server.grpcEndpoint="http://openshell.openshell.svc.cluster.local.:8080"`. |
| Default StorageClass is broken or unsuitable | Deploy `rancher/local-path-provisioner` (v0.0.31 or newer), grant `privileged` SCC to its SAs, `mkdir -p /opt/local-path-provisioner && chmod 1777 && chcon -Rt container_file_t` on every node, mark it default. Without the SELinux `chcon` step the helper pod cannot `mkdir` and PVCs stay `Pending`. |
| Reused hostPath/local data dir | Wipe `/var/openshell-data` (or your custom path) before reinstall, or the gateway dies on the first start with `migration N was previously applied but is missing in the resolved migrations`. |
| Test Kata | Install OpenShift sandboxed containers, verify `RuntimeClass`, create sandbox with Python SDK `SandboxTemplate.runtime_class_name`. |
| Air-gapped image pulls | Mirror gateway, supervisor, sandbox, Agent Sandbox, cert-manager, local-path-provisioner, busybox helper, and operator images with `skopeo copy --all`; override Helm values; attach pull secret to gateway service account and namespace `default` service account. |
| mTLS CLI access | Use cert-manager experiment or built-in PKI on non-OpenShift; manually extract `openshell-client-tls` into CLI gateway `mtls/`. |
| Team isolation | Use one namespace and one Helm release per team. Re-apply the PSA label and SCC binding per namespace, and override `server.grpcEndpoint` per release to match the per-namespace service FQDN. |
| Daemon agents | Run post-ready `openshell sandbox exec` (stdin-piped, non-TTY) or track PR #1326; true declarative daemon startup remains a gap. |
| Declarative/GitOps | Manage Helm release declaratively; do not manage sandbox CRs declaratively for production yet. |

## Release and PRD Notes

- The README still states alpha "single-player mode", which is consistent with the M3 gaps.
- The latest stable release observed during review was `v0.0.43`, published May 18, 2026. The OCI Helm chart followed with `0.0.44` on May 19, 2026. Pin chart/image/CLI versions to the same release rather than using floating `dev` or `latest`; mixing chart `0.0.44` with CLI `0.0.43` is not validated.
- The CLI install script enforces a `BREAKING_RELEASE_VERSION` threshold (currently `0.0.37`) that requires `OPENSHELL_ACK_BREAKING_UPGRADE=1` when crossing. This is fine for fresh installs but is a friction point when upgrading older workstations.
- The visible public GitHub project `OpenShell Roadmap` exists as project #233 and was updated May 19, 2026, but detailed cards were not accessible through the public web page in this review. Roadmap-labeled issues are therefore the actionable public roadmap source used here.
- Several requirements in the PRD have already drifted from the implementation. The most important drift: the PRD assumes a supervisor DaemonSet/hostPath/chcon flow, while current main uses supervisor OCI image side-loading. Keep the PRD focused on the required outcome rather than the old mechanism.
- The May 20, 2026 live walkthrough added four empirically-discovered gaps (supervisor DNS resolver, `runAsNonRoot`/`USER nvs` conflict, namespace PSA enforce label, hardcoded PVC class/size). These should be merged into the next PRD revision as M1 acceptance criteria rather than left to the install guide.
